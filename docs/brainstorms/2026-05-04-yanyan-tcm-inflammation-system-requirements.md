# 炎炎消防队(Yanyan)中医发物 × 次晨体感系统需求

## Summary

用户每餐拍照得到中医语言(发/温和/平)的食物分类,次晨做两阶段打卡(7 维度盲打卡 + 对照昨日),系统综合饮食、体感、iOS 健康、环境因素,在首屏给出"今日体质:中火,火分 62"的混合表达;Day 14 起渐进显示个性化发物候选清单,Day 30 整合为可下载/分享的"<<体质档案命名占位>>"。

---

## Problem Frame

30-45 岁、体检报告里有 1-2 项异常(血糖偏高、尿酸高、或常年鼻炎)、不愿意吃药的中产正在两难处境里循环:体检后焦虑半年 → 忘掉 → 下次体检再焦虑一次。他们听妈妈泛泛地讲"少吃发物"、医生说"少吃糖",但没人告诉他们**自己这个体质**对应的具体发物是哪几样、明天早饭该吃什么。

市场上的产品要么困在卡路里(薄荷)、要么是抗炎评分但浅薄无趋势(Flammy / Inflammatory Food Identifier)、要么是西方代谢话术不针对中餐(Levels / Veri / Whoop)。**中医诊断 2000 年的"食物→次日体感"闭环逻辑从未被任何产品数据化**,而 30-45 岁中文中产对"上火/发物"的母语级共鸣远超对"DII/CRP"的认知亲近。

用户在做选择的瞬间需要一个红黄绿,在体检的间隔期需要一份会变化的指标趋势,在 30 天后需要一份属于自己的发物档案——这三层时间尺度上的信任锚,目前没有产品同时提供。

---

## Actors

**人/组织角色**

- A1. **中产用户**:已体检异常、想自己改饮食、不愿意吃药的 30-45 岁;唯一付费方;唯一需要被讨好的对象。
- A2. **中医顾问委员会**(v2 起):5-10 位老中医组成的食物分类争议裁定委员会,负责对典籍冲突项投票拍板。

**系统组件**(非 Actor 但贯穿多条 Requirements)

- S1. **食物分类引擎**:基于典籍语料 + LLM 派生(v1)、顾问委员会审核修订(v2)、用户反馈持续 fine-tune(v3 起)。
- S2. **Yan-Score 算法**:综合食物 / 体感 / iOS 健康 / 环境因素,产出主观:上火等级 + 火分。
- S3. **发物回归引擎**:在 Yan-Score 之外另跑一路,扣除环境变量,从 14-30 天数据回归个体发物清单。

## Key Flows

- F1. **Onboarding(注册到首次拍照)**
  - **Trigger:** 用户首次打开 App
  - **Actors:** A1, A4
  - **Steps:**
    1. 反向定位筛选屏:"你最希望解决什么"——选项使用上火/发物/体检异常语言,不出现"减肥"
    2. 近期 7 维度症状频次屏:7 个发物维度(鼻塞/起痘/口干/大便/精神/浮肿/喉咙痒)× 过去一周频率(几乎没/偶尔/经常)的方块矩阵,1 屏完成
    3. 体质 baseline 即视感:基于 step 2,系统给一个临时上火等级先验提示("看起来你近期偏微火/中火")
    4. iOS 健康授权请求屏(在 baseline 之后,信任已建立)
    5. 首次拍照引导:"中午吃饭时拍一张,我会先给你看一眼"
  - **Outcome:** 用户进入主屏,可以拍照;系统已有近期症状先验 + 健康数据通道
  - **Covered by:** R1, R2, R3, R4, R5

- F2. **每餐拍照 → 即时反馈**
  - **Trigger:** 用户在主屏点拍照按钮
  - **Actors:** A1, A3, A4
  - **Steps:**
    1. 拍照 / 从相册选;1-3 秒处理 loading
    2. 结果页:整餐火分(数字)+ 食物条目分类(发/温和/平)+ 典籍/营养学引用
    3. 当下建议:"接下来 1-2 餐建议清淡 / 可以放松"
    4. 用户可标记"误识别"反馈
  - **Outcome:** 这一餐进入数据库;Yan-Score 当日累积值更新
  - **Covered by:** R6, R7, R8, R9

- F3. **次晨两阶段打卡**
  - **Trigger:** 早上 7:30 推送(可自定义)
  - **Actors:** A1, A4, A5
  - **Steps:**
    1. **Step 1 盲打卡**:7 维度全展开,默认全无,有反应的勾选并拉滑块定程度;**不提示前一日勾过哪些**
    2. **Step 2 对照展示**:打卡完成后展示"昨天你勾过 X、Y,今早记为 X' / 无 Y"的对照(只展示不输入)
    3. 揭晓今日 Yan-Score:上火等级(平/微火/中火/大火)+ 火分数字 + 归因 breakdown(饮食 X 分 / 环境 Y 分 / 作息 Z 分)
  - **Outcome:** 当日体感数据干净进入回归数据集;用户拿到上火等级和归因解释
  - **Covered by:** R10, R11, R12, R13, R14

- F4. **Day 14-30 群体先验 → Day 30 个体清单交付**
  - **Trigger:** 用户累计打卡满 14 天
  - **Actors:** A1, S3
  - **Steps:**
    1. **Day 14-29 期间:** "发物与发现"tab 仅展示**群体先验**("和你体质类似的人群,常见发物是…"),并明示"个体定制清单将在 Day 30 生成"
    2. **Day 30:** tab 切换为个体定制清单(**三档语言:高可能 / 中可能 / 不显示低可能;对外不暴露具体百分比**)
    3. **Day 30:** 系统生成"<<体质档案命名占位>>" PDF,含发物清单 + 体质画像 + 30 天 Yan-Score 趋势图 + **免责声明列出未扣除的混杂变量(R20)**
    4. 用户可下载 PDF / 分享朋友圈 / 拿给医生看
  - **Outcome:** Day 30 用户拿到一份"小而准"的个体发物档案;Day 14-29 用群体先验维持期待感,不在小样本上做个体强承诺
  - **Covered by:** R21, R21b, R22, R24, R25

- F5. **体检报告后置上传**
  - **Trigger:** 用户主动从首页或个人中心点击"上传体检报告"
  - **Actors:** A1
  - **Steps:**
    1. 拍照或从相册选体检报告 PDF/截图
    2. OCR 解析血糖/尿酸/CRP/血脂/鼻炎记录等异常项
    3. 系统对照之前的体检(如有)生成异常项变化趋势
  - **Outcome:** 长程结果可验证;季度异常项改善率指标可计算
  - **Covered by:** R19, R20

## Requirements

**Onboarding 与体质 baseline**

- R1. Onboarding 全程使用选项 + 滑块,**禁止文字输入**(体检报告 OCR 除外)
- R2. Onboarding 总步数 ≤ 5 屏(包括首次拍照引导)
- R3. 反向定位筛选屏使用"上火/发物/体检异常"语言,**不出现"减肥""卡路里"字样**
- R4. 近期 7 维度症状频次作为体质 baseline,1 屏完成,采用方块矩阵(7 维度 × 3 档频率)
- R5. **(H5 平台调整)** 健康数据接入:**v1 H5 形态下,放弃 HealthKit / 华为健康 等 native SDK 通道**;改为(a)用户手动同步微信运动步数,(b)允许用户跳过此步,Yan-Score 算法在缺失健康数据时降级运行。授权请求出现在 baseline 完成之后、首次拍照引导之前。
- R5b. **《个人信息保护法》合规节点**:onboarding 中健康/医疗数据采集前(R4 baseline 屏完成时)插入**敏感个人信息单独同意页**,告知数据用途、存储地点、保留期限、用户撤回路径;勾选同意为强制项(不勾选不能进入下一屏)。具体页面文案待法务审核(ce-plan 阶段)

**双闭环输入:拍照 + 次晨打卡**

- R6. 拍照 → 结果页 AI 处理时延目标 **2-5 秒**(H5 形态下网络往返较 native 慢),>5 秒显式 loading 状态 + 进度提示
- R7. 食物分类输出三档:发(红)/ 温和(黄)/ 平(绿);UI 上每个食物条目附中医典籍引用或现代营养学引用
- R8. 结果页**不使用 AI 人格化主播 / 营养师角色**(无"林医生说"叙事)
- R9. 用户可对 AI 食物识别结果标记"误识别",反馈进入分类引擎修订队列
- R10. 次晨打卡分两阶段:Step 1 盲打卡 7 维度全展开,Step 2 对照展示昨日勾选
- R11. Step 1 中 7 个维度默认状态均为"无";用户主动勾选"有反应"后,出现该维度专属严重度滑块(每维度档位独立设计,3-5 档,非 100 分制)
- R12. Step 1 **不展示**前一日的打卡数据,以避免锚定偏倚污染发物回归数据集
- R13. Step 2 仅作展示,不再要求用户输入
- R14. 当日 Yan-Score 在 Step 2 之后揭晓,以防体感数据被分数预设影响

**Yan-Score(核心指标)**

- R15. Yan-Score 由两部分组成:上火等级(平/微火/中火/大火,4 档)+ 火分数字(0-100)
- R16. 首屏以上火等级为视觉主体,火分数字与周内趋势作为辅助
- R17. Yan-Score 算法综合 **3 类核心输入**:饮食、体感、环境(花粉/PM2.5/季节);**iOS 健康 / 微信运动步数等作为可选辅助输入**,缺失时算法降级运行(权重重分配,不影响展示)。算法必须设计为**支持任意子集缺失**
- R18. 用户在 Yan-Score 旁可点击查看归因 breakdown(饮食 X 分 / 环境 Y 分 / 作息 Z 分)
- R19. 首次 Yan-Score 在 Day 2 早晨完成首次次晨打卡之后揭晓(Day 1 不展示 Yan-Score)
- R20. **后台另跑一路扣除环境变量的回归算法**用于发物清单计算,与展示用 Yan-Score 算法分离。**已知不扣除的混杂变量(列入 PDF 免责声明):月经周期 / 急性感冒 / 工作压力期 / 出差水土 / 酒精摄入 / 熬夜**——这些变量超出 v1 数据采集能力,女性用户经期前后的发物归因结果需用户自行交叉判断。
- R20b. **周内趋势线最小展示阈值** = 累计打卡满 21 天;< 21 天首屏只展示当日上火等级 + 火分,不画趋势线,文案"数据累积中,先不画趋势"。

**发物清单与个人体质证**

- R21. **累计打卡满 14-29 天期间**,"发物与发现"tab 仅展示**群体先验**:"和你体质类似的人群,常见发物是…",并明示"个体定制清单将在累计 30 天后生成"
- R21b. **累计打卡满 30 天起**,"发物与发现"tab 切换为**个体定制清单**——基于个体 30 天数据 + 群体先验做 Bayesian shrinkage,显示候选发物
- R22. 候选发物清单**默认仅显示高/中可能性两档**(三档分类:高可能 / 中可能 / 不显示低可能);**对外不暴露具体置信度数字**(如"75%"),仅用三档语言描述。低于阈值的项保留在算法内部,不展示
- R23. 用户可对系统判定的某项发物提交"我吃了没事"反例:**通过"选择已识别的食物 → 标记反例"两步操作完成,不允许文字输入**;反例进入个性化算法 update
- R24. Day 30 生成"<<体质档案命名占位>>"PDF,含发物清单 + 体质画像 + 推荐饮食 + 30 天 Yan-Score 趋势图
- R25. 个人体质证 PDF 可下载 / 分享 / 截图(无 DRM 水印强制)
- R26. 名称"<<体质档案命名占位>>"为占位,正式发布前需做名称命名研究

**食物分类引擎(v1 范围)**

- R27. v1 食物分类引擎:基于中医典籍(《本草纲目》《温病条辨》等)语料 + LLM 自动派生;**v1 上线后到顾问委员会到位之间(乐观 3 个月,实际可能 6-12 个月)的争议裁定机制需要在 ce-plan 阶段定义**(候选:产品/工程团队人工 + 流派标签;或暂缓发布有争议条目)
- R28. 用户反馈机制始终在线:误识别(R9)+ 反例(R23)两个通道持续 fine-tune 引擎

## Acceptance Examples

- AE1. **Covers R10, R11, R12.** Given 用户昨日 onboarding 时勾过"鼻塞""口干"作为近期症状,when 次晨 7:30 推送他打卡,Step 1 界面 7 个维度全部默认"无"、且不显示昨日勾选信息;用户主动勾选"鼻塞"后该维度滑块出现,他选了"轻度";用户未勾选"口干"。
- AE2. **Covers R13, R14, R18.** Given AE1 的 Step 1 完成,when 进入 Step 2,展示"昨天你勾过鼻塞、口干;今早:鼻塞 → 轻度,口干 → 无"的对照;之后揭晓"今日体质:微火,火分 38";用户点击火分,看到 breakdown:"饮食 22 / 环境 10(今日花粉中等)/ 作息 6"。
- AE3. **Covers R19.** Given 用户**累计打卡** 1 天(无次晨打卡),when 当日晚上打开 App,主屏不显示火分/上火等级,而是显示"明早打卡后揭晓你的首份火分"。
- AE4. **Covers R21, R21b, R22.** Given 用户**累计打卡满 20 天**(Day 14-29 期间),when 用户进入"发物与发现"tab,看到的是"和你体质类似的人群常见发物"群体先验仪表 + "个体定制清单将在 Day 30 生成"提示;Given 用户**累计打卡满 30 天**,when 进入同一 tab,看到的是个体定制清单,以三档语言("高可能"/"中可能")呈现,**不出现具体百分比数字**。
- AE5. **Covers R23.** Given 系统判定某用户的"海鲜"为发物(达到 ≥60% 阈值),when 该用户对"海鲜"条目标记 3 次反例,系统对该用户的海鲜判定置信度下调,可能跌出阈值。
- AE6. **Covers R20, R20b.** Given 用户连续累计打卡满 30 天 + 拍照,when 系统运行,Then 同时输出展示用 Yan-Score(全综合,含环境)和扣环境后的发物候选清单,用户在前端看到的火分归因 breakdown 与发物清单结论**互不推导**,前端文案明示"两套指标服务不同目的"。

## Success Criteria

**用户结果**

- Day 30 时,用户能口头说出至少 3 样属于自己的发物
- 90 天季度异常项改善率(主动上传两次体检报告的用户中,至少 1 项异常改善的占比)≥ 25%
- 用户主动把"个人体质证 PDF"分享给家人或医生的占比 ≥ 30%

**留存与产品健康**

- 周活跃记录率(WAR,过去 7 天 ≥5 天每天 ≥2 餐拍照 + 完成次晨打卡):8 周后稳定在 ≥ 35%
- 次晨打卡完成率(当日有食物记录的用户中次日完成打卡的占比):≥ 60%
- 个体发物清单生成率(**累计打卡满 30 天**的用户中已生成 ≥3 项个体发物高/中可能项的占比):≥ 70%
- 减肥目的用户占比 < 30%(反向指标;通过新用户问卷 + 6 个月留存按目的分群反查双通道监测)

**下游交接信号**

- 进入 ce-plan 时,产品团队不再需要回答"中医辨证维度选哪些""权威叙事用什么人格""次晨打卡如何避免锚定偏倚""发物清单何时交付以何种形式"等产品决策
- ce-plan 阶段需要研究/选择的应集中在:LLM 食物识别模型选型、Yan-Score 加权公式、Bayesian prior 数据来源、PDF 生成管线、iOS 健康 API 调用细节

## Scope Boundaries

- **不做卡路里/血糖曲线/西方代谢健康年龄叙事** —— 让出给 Levels / Veri / 薄荷
- **不做西方"抗炎科学权威"叙事**(hs-CRP/DII 不作为产品 talking point)—— 后台保留映射,不打招牌
- **不做保险/医疗精算变现路径**(Phase 1)—— 与中医叙事的循证标准存在结构性冲突,若未来要做需另起科学版产品线
- **不做健身教练/私教课程**
- **不做保健品/电商**
- **不服务儿童/老人/孕妇等特殊人群** —— 医学风险高、获客模式完全不同
- **不在 onboarding 里跑国标 9 体质完整问卷** —— 选了简化版近期症状频次,代价(放弃国标背书 talking point + baseline 是状态而非长期 trait)被接受
- **不做 AI 人格化主播角色**("林医生"风格)—— UI 上无人格,采用典籍引用 + 数据简语
- **不在 Day 0 即时给 Yan-Score** —— 必须等 Day 2 首次次晨打卡
- **不在打卡/拍照/体检上传以外的环节使用文字输入**
- **不做多种推送频率** —— 每天 1 次次晨打卡推送 + 拍照后即时反馈,不做多个时段提醒

## Key Decisions

- **Yan-Score 作为核心指标(C 选择)**:接受日活机制清晰但"和 Levels/Veri 在同轨道竞速"的代价;差异化由"中医叙事 + 拍照零硬件"两点支撑。STRATEGY.md 已相应更新。
- **次晨打卡两阶段(盲打 + 对照)**:多花 15 秒换数据干净度,因为发物清单是最贵的资产,污染不起。
- **Yan-Score 全综合 + 后台另一路扣环境**:前者满足用户体验(归因感),后者满足算法正确性(扣除混杂变量才能 attribute 到食物)。两套数据流并存。
- **权威叙事:典籍引用 + 数据简语,无人格**:失败时不需要换 AI 人格,UI 长期可维护;代价是早期"温度感"弱。
- **Onboarding baseline 用近期症状频次而非国标 9 体质**:1 屏完成 vs 5 屏 60 题,转化率优先;代价是放弃国标 talking point。
- **发物清单 = 渐进式 + PDF 体质证**:不靠仪式感,靠累积可见和可分享性建立留存,与"渐进式置信暴露"算法可信度策略匹配。
- **食物分类引擎走 v1→v3 路线图**:v1 典籍打底 + LLM 派生(R27);v2(预计 6-12 个月后)引入 5-10 位老中医组成的顾问委员会对争议项审核修订,**输出应为流派标注的多分类(如"经方派认为发,温病派认为温和"),而非单一裁定**;v3 中期目标与某中医药大学联名背书并合作出产论文 / 数据库。接受 v1 学术严肃性弱的代价。
- **平台 = H5(v1)**:跨平台一击覆盖 iOS / Android,绕开"安卓 60% 默认排除"的反向定位漏洞;接受三个代价 —— 健康数据通道弱(无 HealthKit / 华为健康 SDK)、推送受限(无后台主动推送,只能模板消息)、拍照速度较 native 慢。Yan-Score 算法因此降级到 3 类核心输入(饮食 / 体感 / 环境),iOS 健康为可选辅助。Native app 列入 v2/v3 路线图。
